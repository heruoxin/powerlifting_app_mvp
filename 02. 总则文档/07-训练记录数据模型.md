# 训练记录数据模型

## 文档目的

本文档用于把当前已经收敛下来的训练记录与训练执行设计，整理成更贴近实现的数据模型约束。

本文档当前基于 [数据结构设计-训练记录.yml](/Users/heruoxin/dev/PRD/电子教练/01.%20人工文档/数据结构设计-训练记录.yml) 的早期思路整理，但已按最新 `Tab02` 设计重新收敛。

## 核心结论

当前最重要的结构结论如下：

- 训练记录不再以“平铺节点流 + 显式 `rest` 节点”为核心表达。
- 训练记录更适合被理解为：`训练会话元数据 + 有序动作块列表 + 动作块内有序 set 列表`。
- 训练负荷的最小记录单位仍然是单组 `set`。
- `training_record` 层级下恢复显式 `state` 字段，用于表达整场训练会话的生命周期终态。
- 单组数据不再只保留 `plan + actual` 两层，而是改为：
  - `baseline_plan`
  - `working_plan`
  - `actual`
- `baseline_plan` 冻结在用户点击“开始训练”的那一刻，作为练后默认差异对比的唯一计划基线。
- `working_plan` 只服务训练中执行，不参与练后默认 `计划差异` 的比较。
- 所有 `uid` 统一使用随机十六进制值，不使用 `session-001`、`node-001` 这类带语义前缀的字符串。
- 普通组间休息不再作为独立 `rest` 节点出现，而是由相邻组的时间戳差值自然推导；必要时只额外记录显式暂停事件。
- 为适配当前页面中的动作卡、列配置与推测列，动作块应成为数据层中的一等对象，而不是纯展示层临时聚合结果。
- `随便练练` 在模型层统一命名为 `Open Session`，与 `D1 / D2 / D3` 属于同级训练槽位，使用 `OS1 / OS2 ...` 编号。

## `uid` 规则

### 统一规则

- `training_record.uid`
- `exercise_block.uid`
- `set.uid`
- `pause_event.uid`

以上 `uid` 全部统一使用随机十六进制值。

示例：

- `8b4e07a5fc39`
- `0ac3665f927b`
- `71a0cc8e12bd`

补充说明：

- 首版建议统一使用固定长度随机十六进制字符串，例如 `12` 位。
- `uid` 只负责唯一标识，不承担排序、可读命名或业务含义。

## 记录层级

当前建议将训练记录理解为三层：

1. 训练会话级元数据
2. 动作块列表
3. 动作块内的单组 `set` 列表

## 第一层：训练会话级元数据

用于描述“一次训练”本身，而不是描述某一组动作。

### 建议字段

- `uid`
  - 训练记录唯一标识。
- `date`
  - 训练日期。
- `state`
  - 训练会话状态。
- `ended_reason`
  - 训练结束原因。
- `day_label`
  - 用户可见的周期标签，例如 `W3D2`、`W3OS1`。
- `week_index`
  - 当前属于第几个 `W`，从 `1` 开始。
- `day_slot_type`
  - 训练槽位类型。
- `day_slot_index`
  - 槽位序号，例如 `D2` 中的 `2`，或 `OS1` 中的 `1`。
- `day_slot_label`
  - 槽位标签，例如 `D2`、`OS1`。
- `source_plan_day_uid`
  - 若来自既定计划，则记录对应训练日 `uid`；若来自 `Open Session`，则为空。
- `started_at`
  - 本次训练开始时间。
- `finished_at`
  - 本次训练结束时间。
- `exercise_blocks`
  - 有序动作块列表。
- `pause_events`
  - 显式暂停事件列表，可选。

### `training_record.state`

当前建议显式保留以下枚举：

- `in_progress`
- `completed`
- `terminated`

说明：

- `training_record.state` 用于表达整场会话是否仍在进行、是否正常完成、是否因规则或异常被终止。
- `set.state` 继续用于表达组级执行状态，两者不再承担同一层语义。
- 顶层会话状态不再强行由各组状态反推。

### `ended_reason`

首版建议至少支持：

- `completed_by_user`
- `terminated_timeout_8h`

说明：

- `completed_by_user` 表示用户主动点击完成训练。
- `terminated_timeout_8h` 表示系统在启动 App 时发现该会话仍为 `in_progress`，且距离 `started_at` 已超过 `8` 小时，因此自动判定训练终止。

### `day_slot_type`

当前建议至少支持：

- `planned_day`
- `open_session`

说明：

- `planned_day` 对应 `D1 / D2 / D3 ...`
- `open_session` 对应 `OS1 / OS2 ...`
- `Open Session` 比 `Easy Train` 更适合作为英语对象名，因为它表达“未绑定既定计划的开放训练”，不会误导成“轻松训练 / 低强度训练”。

### 训练会话状态与 App 启动时的自动终止规则

- 只要存在 `training_record.state = in_progress` 的会话，App 每次启动时都应检查：
  - 当前时间与 `started_at` 的差值是否超过 `8` 小时
- 若超过 `8` 小时：
  - 自动将该会话改写为 `state = terminated`
  - 自动补写 `finished_at = 当前检查时间`
  - 自动补写 `ended_reason = terminated_timeout_8h`

展示层补充规则：

- 当 `training_record.state = completed` 或 `terminated` 时，仍处于 `planning` 的原计划组，展示层统一解释为 `未执行`
- 当 `training_record.state = completed` 或 `terminated` 时，`skipped` 仍展示为 `跳过`
- 因此 `未执行` 不再要求额外新增一套组级状态枚举，而是由“会话已结束 + 该组仍未开始”共同推导

### 训练会话与计划来源的关系

- 若一次训练来自既定训练日，则训练会话应记录 `source_plan_day_uid`。
- 若一次训练来自 `Open Session`，则 `source_plan_day_uid` 为空。
- 一次训练在点击“开始训练”时，应对当时的计划内容做一份冻结快照，写入每一组的 `baseline_plan`。
- 训练记录中的单组数据改为保留：
  - `baseline_plan`
  - `working_plan`
  - `actual`
- 训练中若用户修改未来未完成组，这些变化只写入当前组的 `working_plan`。
- 若该组是在训练中被改过未来执行方案，则通过 `working_plan_edited_during_session = true` 标记。
- `actual` 只记录真实完成值。

这样在产品层才能支持：

- 训练完成后默认比较 `baseline_plan vs actual`
- 训练中继续编辑 `working_plan`，而不污染练后差异对比基线
- 对 `本次新增` 与 `训练中调整` 给出明确标签

补充说明：

- `working_plan` 仍应被保留在训练记录中，用于回看训练中临时调整过的执行方案。
- 但完成页顶部的 `计划差异` 默认不比较 `working_plan`，而是始终比较 `baseline_plan` 与 `actual`。
- 若一组原本来自 `baseline_plan`，但训练中被用户从后续执行计划里移除，则保留该组的 `baseline_plan`，并将 `working_plan` 置空；这样练后仍可稳定显示该组 `未执行`。

## 语言与本地化值

- 所有用户可见的语言相关值，应使用用户设备语言或用户语言偏好的习惯值。
- 这包括但不限于：
  - 动作名称
  - 动作备注
  - 组次备注
  - 推测项名称
  - 用户可见的单位标签

例如：

- 若设备语言为中文：
  - `深蹲`
  - `硬拉`
  - `卧推`
  - `kg`
  - `lb`
  - `自重`
- 若设备语言为英文，则使用对应英文值。

补充说明：

- 结构字段名本身仍可保持稳定代码风格。
- 这里约束的是“写入与展示给用户的语言相关值”，而不是 JSON / YAML 键名。

## 第二层：动作块列表

动作块不再只是 UI 临时聚合结果，而应成为训练记录中的正式层级。

原因：

- 当前 `Tab02` 的三种状态页都以动作卡为核心结构。
- 支持动作拖拽排序。
- 支持动作级设置对话框。
- 支持按动作保存显示列配置与推测配置。

### 列表规则

- `exercise_blocks` 是一个有序数组。
- 数组顺序就是页面显示顺序与默认执行顺序。
- 动作拖拽排序的结果，直接反映在该数组顺序变化上。

### 每个动作块建议字段

- `uid`
- `name`
  - 动作名称，例如“卧推”“泽奇深蹲”。
- `exercise_category`
  - 动作分类。
- `source_type`
  - 例如：
    - `planned`
    - `session_added`
- `display_columns`
  - 当前动作卡显示哪些列。
- `derived_metric_config`
  - 当前动作卡启用了哪些推测内容。
- `note`
  - 动作级备注，可选。
- `sets`
  - 该动作块内有序 `set` 列表。

### `exercise_category`

当前建议与后台动作元数据表保持一致，首版至少包含：

- `主项`
- `主项变式`
- `辅助项`
- `有氧运动`

### `display_columns`

该字段用于保存当前动作卡的列配置快照，例如：

- `load_value`
- `load_unit`
- `duration`
- `duration_unit`
- `rep`
- `rpe`
- `note`
- `derived`

说明：

- 动作元数据表负责提供默认配置。
- 训练记录自身应保存动作卡当时使用的列配置快照。
- 这样即便后台动作库默认配置 later 改变，历史记录仍能按当时配置稳定渲染。

### `derived_metric_config`

该字段建议使用对象结构，而不是直接使用平铺数组。

首版先只约束到这一层：

```yaml
derived_metric_config:
  items: []
```

说明：

- `items` 用于表示当前动作卡启用了哪些推测项。
- `items` 内部的 item 结构当前暂不细化，为后续扩展留出空间。
- 这里保存的是“要展示什么”，不是计算结果本身。
- 具体推测值仍应根据当前用户数据与当前行数值实时计算。

## 第三层：单组 `set` 列表

单组 `set` 仍然是训练负荷记录的最小单位。

### 列表规则

- `sets` 是动作块内部的有序数组。
- 数组顺序就是页面中的组序。
- 组内拖拽排序的结果，直接反映在该数组顺序变化上。

### 每个 `set` 建议字段

- `uid`
- `state`
- `source_type`
  - 例如：
    - `planned`
    - `session_added`
- `working_plan_edited_during_session`
  - 是否在训练中修改过未来执行方案。
- `started_at`
- `finished_at`
- `baseline_plan`
  - 点击开始训练瞬间冻结下来的原计划值。
- `working_plan`
  - 当前这组在本次训练中的执行计划值。
- `actual`
  - 实际完成值。
- `effort_metrics`
  - 实际主观强度等指标。

### `set.state`

当前建议保留以下枚举：

- `planning`
- `in_progress`
- `completed`
- `skipped`

说明：

- `set.state` 只负责组级执行状态。
- 训练会话顶层的 `training_record.state` 单独存储。
- 动作块状态若需要展示，可继续由其内部 `set.state` 推导。

### `source_type`

- `planned`
  - 原本就属于点击开始训练瞬间冻结快照中的一组。
- `session_added`
  - 训练中新增的一组。

这对应页面中的：

- `原计划`
- `本次新增`

### `working_plan_edited_during_session`

- 若一组原本来自冻结快照，但在训练中改过未来执行方案，则该字段为 `true`。
- 这对应页面中的：
  - `训练中调整`

## 负荷表达方式

旧版文档使用 `weight / weight_unit`，已经不完全适应当前设计。

原因：

- 页面里有时显示的是 `重量`
- 有时显示的是 `强度`
- 单位还可能切成 `%` 或 `RPE`
- 某些动作则会出现 `自重` 等非数值表达

因此，当前更建议把这一层统一抽象为 `load`。

### 建议字段

在 `baseline_plan`、`working_plan` 与 `actual` 中，优先使用：

- `load_value`
- `load_unit`
- `load_text`
- `rep`
- `duration`
- `duration_unit`
- `distance`
- `distance_unit`
- `note`

### 字段说明

#### `load_value`

- 用于数值型负荷。
- 仍支持范围表达。

#### `load_unit`

首版至少支持：

- `kg`
- `lb`
- `%`
- `RPE`

#### `load_text`

用于无法用简单数值表达的负荷，例如：

- `自重`
- `辅助`
- `弹力带`

说明：

- 当使用 `load_text` 时，`load_value` 与 `load_unit` 可为空。

### 数量字段的范围语法

首版继续使用以下表达：

- `[90]`
  - 表示单值，例如 `90kg`
- `[90, 100]`
  - 表示区间，例如 `90~100kg`
- `[90, null]`
  - 表示下限，例如 `90kg+`
- `[null, 90]`
  - 表示上限，例如 `90kg-`

### `rep` 的展示语义

- `[5]`
  - 表示固定次数 `5`
- `[8, 10]`
  - 表示范围次数 `8~10`
- `[10, null]`
  - 表示下限次数 `10+`

训练完成后，`actual.rep` 只记录单值，不保留范围。

### `RPE` 的两层语义

这里需要明确区分两种不同含义：

- `load_unit = RPE`
  - 表示计划值用目标强度表达，而不是绝对重量。
- `effort_metrics.rpe`
  - 表示训练中或训练后记录的实际主观强度。

这两者不能混为同一个字段。

## `actual` 与 `effort_metrics`

### `actual`

用于记录真实完成值。

约束如下：

- 仅在单组已完成时存在。
- 所有数量字段都应落为单值或空值。
- `actual` 不记录范围。

### `effort_metrics`

用于记录训练表现与主观指标。

当前已知至少包含：

- `rpe`

后续若需要扩展速度、疼痛、主观恢复等信息，可继续增加，但不应和 `actual` 混写。

## 休息与暂停的记录方式

### 不再使用显式 `rest` 节点

当前已不建议把组间休息建模为独立 `rest` 节点或 `rest` 块。

原因：

- 当前页面设计中不存在休息块。
- 训练中的休息更接近页面状态，而不是独立训练项目。

### 普通组间休息

- 普通组间休息由相邻两组的时间戳差值自然推导：
  - 上一组 `finished_at`
  - 下一组 `started_at`

### 显式暂停

若需要记录长时间中断、切后台或手动暂停，可在训练会话层增加 `pause_events`：

- `uid`
- `pause_type`
- `started_at`
- `finished_at`
- `description`

首版 `pause_type` 可包含：

- `manual_pause`
- `background_pause`

## 顺序与展示关系

当前建议采用以下区分：

- 数据层：
  - `training_record`
  - `exercise_blocks[]`
  - `sets[]`
- 展示层：
  - 直接使用动作卡展示 `exercise_blocks`
  - 使用表格行展示 `sets`
- 交互层：
  - 动作拖拽直接改 `exercise_blocks` 顺序
  - 组序拖拽直接改 `sets` 顺序

这样可以同时满足：

- 数据层与 UI 层结构一致
- 拖拽排序不需要额外转换
- 训练中与训练后对比更直观

## 推测列与动作元数据表的关系

当前已知，训练记录不能只保存数值，还需要保存“怎么展示这些数值”。

因此建议产品保持一份后台动作元数据表，每条记录至少包含：

- `canonical_name`
- `aliases`
- `exercise_category`
- `default_columns`
- `supported_units`
- `supported_derived_metrics`
- `reference_lift_key`
- `default_row_template`

说明：

- 动作元数据表负责提供默认列配置与默认单位。
- 训练记录里的动作块保存当前配置快照。
- `derived_metric_config` 保存“需要展示哪些推测内容”。
- `derived_metric_config` 首版建议统一保存为 `{ items: [] }` 对象结构。
- 推测结果本身不建议作为持久化真相源存储，而应按当前数据实时计算。

## 历史记录修改原则

### AI 权限

- 训练完成后，AI 不可修改历史训练的结构化数据。
- AI 可以基于历史记录生成总结、观察和建议，但不应直接改写记录内容。

### 用户权限

- 训练完成后，用户仍可修改历史训练记录。
- 用户点击保存后，不保留持久化版本历史。
- 若需要编辑过程中的回退体验，应由 UI 层提供撤销与重做能力，而不是由数据层长期保存版本链。
