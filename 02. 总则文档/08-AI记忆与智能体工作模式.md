# AI记忆与智能体工作模式

## 文档目的

本文档用于定义本产品中的 AI 教练应借鉴怎样的智能体行为模式，以及本地 Markdown 记忆体系如何设计。

这里参考的是你提到的 openClaw 式模式，但本文档关注的是“产品化后的行为要求”，而不是复制某个具体项目的实现细节。

## 对 openClaw 式模式的当前理解

基于你的描述，我认为这种模式的核心不在于某个具体框架，而在于三类能力：

### 1. 任务拆分与多步工具调用

- AI 不应把复杂问题当成一次性问答。
- 它应能先拆成若干子任务，再针对每个子任务调用工具。
- 当某个子任务还有可用工具时，应继续调用，直到阶段性信息足够，再回到更高层任务继续判断。

放到本产品里，这意味着：

- 讨论计划调整时，AI 可以先读当前周期、再读最近训练记录、再读用户特质，最后形成建议。
- 讨论动作问题时，AI 可以先读最近训练记录、再读相关笔记、再结合训练理论知识库组织回答。

### 2. 多层记忆，而不是单一上下文

- AI 不应只依赖当前会话上下文。
- 它需要有相对稳定、可持续维护的本地 Markdown 记忆体系。
- 不同文件应承担不同层级的职责。

### 3. 通过受控 skill 操作产品对象

- AI 的行动能力不应建立在通用 shell 或系统工具上。
- 在本产品中，AI 应通过训练计划、训练记录、训练笔记等对象的受控 skill 完成读写与分析。

## 本产品中的智能体行为要求

结合本项目，我建议将 AI 教练的工作模式定义为：

- 以任务拆分驱动复杂教练工作。
- 以本地 Markdown 记忆维持长期连续性。
- 以受控 skill 访问结构化训练数据。
- 以触发机制驱动后台总结、观察和建议生成。
- 以自动触发话题把后台 trigger 结果外显为可继续追问的用户对话。

## 记忆体系设计原则

### 原则 1：`soul.md` 保持稳定

`soul.md` 更适合作为教练的核心目标、角色设定、工作原则和语气边界。

因此它应：

- 在每次任务中加载。
- 保持相对稳定。
- 不因为单次训练波动而频繁改写。

这与 openClaw 式“日记汇总后回写核心记忆”的精神一致，但在本产品里不建议把所有阶段性观察都直接写回 `soul.md`，否则会把“教练身份”和“用户状态”混在一起。

### 原则 2：用户相关信息应与教练身份分离

用户的身体状态、偏好、习惯、伤病、恢复特点等，不应直接塞进 `soul.md`。

更合适的做法是单独维护用户相关文件。

### 原则 3：日记负责事件沉淀，不负责长期定稿

`日记.md` 应承担事件流记录职责：

- 每次完成训练后追加。
- 每次完成一次重要的 AI 教练沟通后追加。
- 记录当时发生了什么、AI 如何判断、后续关注点是什么。

但它不适合作为每次都完整加载的主记忆文件，否则会不断膨胀。

### 原则 4：应存在一层“从日记提炼出的中层结论”

如果只有 `soul.md` 和 `日记.md` 两层，信息要么过于稳定，要么过于原始。

因此我建议在你提出的文件基础上，再增加一份更适合沉淀中期结论的文件。

## 推荐的记忆文件划分

当前已确认，本产品采用以下五类 Markdown 文件：

### 1. `soul.md`

职责：

- 存储教练的目标任务。
- 存储教练的性格设定与语气边界。
- 存储教练的工作原则、风险偏好和通用行为规范。

特点：

- 每次任务默认加载。
- 变更频率最低。
- 更像“核心教练人格与使命”。
- 在产品运行态视为只读，不作为用户可编辑文件。

### 2. `训练计划.md`

职责：

- 存储当前训练周期所处阶段。
- 记录当前周期的设计思路。
- 记录近期计划安排的核心意图与调整逻辑。

特点：

- 在涉及计划、周期、强度安排时优先加载。
- 可由 AI 在用户确认后的计划变更基础上更新。
- 更像“当前周期的教练计划说明书”。

### 3. `用户特质.md`

职责：

- 存储用户的身体状态、偏好、习惯、伤病、恢复特征等。
- 记录相对稳定但会缓慢变化的用户画像。

特点：

- 适合承接从多次训练和长期观察中提炼出的稳定结论。
- 不应写入太细碎的事件流。

### 4. `教练观察.md`

职责：

- 存储近期到中期的观察结论、趋势判断、待验证假设。
- 连接原始日记与相对稳定的用户特质/计划思路。

特点：

- 比 `日记.md` 更浓缩。
- 比 `soul.md` 更贴近当前用户状态。
- 很适合作为 `训练笔记页 > 教练笔记` 的核心展示来源之一。

### 5. `日记.md`

职责：

- 作为事件型追加日志。
- 记录每次训练完成后的总结条目。
- 记录每次重要 AI 教练沟通后的总结条目。

特点：

- 以追加为主。
- 不追求高度结构化，但建议保持固定模板。
- 作为后续更新 `训练计划.md`、`用户特质.md`、`教练观察.md` 的原始素材。

## 记忆文件之间的关系

建议关系如下：

```text
soul.md
  -> 定义教练是谁、如何工作

训练计划.md / 用户特质.md / 教练观察.md
  -> 定义当前用户与当前阶段是什么状态

日记.md
  -> 记录每次重要事件发生了什么
```

进一步可以理解为：

- `soul.md` 是最稳定的顶层教练记忆。
- `训练计划.md`、`用户特质.md`、`教练观察.md` 是中层工作记忆。
- `日记.md` 是底层事件流记忆。

## 记忆更新机制

### 1. 每次重要事件后写入 `日记.md`

当前建议至少包括：

- 完成一次训练后。
- 完成一次重要的 AI 教练沟通后。
- 周期阶段结束后。

### 2. 定期从 `日记.md` 提炼更新中层记忆

建议优先更新：

- `训练计划.md`
- `用户特质.md`
- `教练观察.md`

这里是我对 openClaw 模式的一个本地化改造：

- 不把日记中的阶段性内容频繁回写到 `soul.md`
- 而是优先回写到更贴近用户状态的中层记忆文件

这样可以避免 `soul.md` 被污染成“教练身份 + 用户近况 + 临时观察”的混合体。

### 3. `soul.md` 仅在较高层规则变化时调整

例如：

- 产品定义的教练角色变化。
- 通用的教练风格与安全边界变化。
- 产品级的行为模式变化。

### 4. trigger 自动介入时，先写记忆再建对话

当 trigger 触发 AI 自动介入时，建议采用固定顺序：

1. 读取必要的本地与云侧上下文。
2. 优先更新 `日记.md`、`教练观察.md`，必要时再更新 `训练计划.md` 或 `用户特质.md`。
3. 在 `AI教练页` 中生成自动触发话题，向用户外显这次介入结果。
4. 若异常程度很高，再由客户端追加本地通知，把用户引导到该话题。

这样可以保证：

- 记忆先被整理，后续任何对话都有稳定依据。
- 自动触发对话不是脱离记忆体系的一次性弹窗。
- 严重异常时，用户既能收到提醒，也能直接看到 AI 依据什么判断。

## 记忆加载建议

### 默认加载

建议每次任务默认加载：

- `soul.md`
- `用户特质.md`

### 按场景加载

- 计划讨论：重点加载 `训练计划.md`
- 长期趋势讨论：重点加载 `教练观察.md`
- 回溯事件：读取 `日记.md` 的相关片段，而不是默认全量加载

这样更适合控制上下文长度，也更接近真正可落地的产品实现。

## 与 `教练笔记` 的关系

当前已确认：

- `训练笔记页 > 教练笔记` 是向用户暴露 AI Markdown 记忆文件的正式入口。

结合上述划分，我的建议是：

- `教练笔记` 优先展示 `日记.md` 与 `教练观察.md` 中适合外显的内容。
- `soul.md` 更偏系统级教练设定，不一定需要默认直接暴露。
- `训练计划.md` 与 `用户特质.md` 是否默认对用户开放，可再根据产品透明度需求决定。

当前已确认的展示规则：

- `教练笔记` 默认展示 `日记.md` 与 `教练观察.md`。
- `soul.md`、`训练计划.md`、`用户特质.md` 不默认直接展示。
- 上述非默认文件可由用户手动展开查看。
- 其中 `soul.md` 仅支持查看，不支持编辑。

## 与服务端和 tool 的关系

- 记忆文件默认保存在本地。
- AI 调用时，可按任务需要把必要片段发往服务端，再由服务端注入 skill 与知识资源。
- AI 可使用的工具应聚焦训练计划、训练记录、训练笔记等产品对象。
- 不向 AI 暴露通用 shell 或系统级工具。
